# 关键决策记录

记录项目中的重要技术选型和设计决策，避免重复讨论。

## 1. GUI 框架：PySide6

**决策时间**：2026-04-29

**选择**：PySide6（放弃 PyQt6）

**原因**：
- PySide6 是 LGPL 协议，公司闭源使用免费
- PyQt6 是 GPL 协议，商业闭源需要购买授权
- 两者 API 几乎相同，迁移成本低

## 2. 数据库：PostgreSQL

**决策时间**：2026-04-29

**选择**：PostgreSQL（放弃 SQLite）

**原因**：
- 公司有多名员工在同一局域网使用，需要共享图库
- SQLite 是文件级锁，不支持多用户并发写入
- PostgreSQL 支持局域网多客户端同时访问
- PostgreSQL 的全文搜索功能可以支持中文搜索

## 3. 分类体系：二级结构

**决策时间**：2026-04-29

**选择**：项目类型 → 风格（二级分类）

**原因**：
- 室内设计公司的项目通常按"项目类型"（如售楼处、酒店、办公室）和"风格"（如现代极简、新中式）分类
- 导入路径：`STORAGE_ROOT/{项目名}/{项目类型}/{风格名}/`
- 项目名仅用于文件夹路径，不参与分类（避免项目名重复导致分类混乱）

## 4. 砍掉图片去重功能

**决策时间**：2026-04-29

**选择**：不实现图片去重

**原因**：
- 首次扫描五万张图片时，去重计算会严重拖慢速度
- 用户可以手动管理重复图片
- 减少首次使用门槛

## 5. 线程模型：所有 DB IO 在 QThread 中执行

**决策时间**：Phase 4 开发期间

**选择**：所有数据库操作必须在 QThread 中执行，禁止在主线程阻塞

**原因**：
- PySide6 的主线程是 UI 线程，阻塞会导致界面卡死
- 使用 QueryWorker + run_query 便捷函数，统一异步执行
- 停止 Worker 前必须先断开信号，防止旧信号污染新 Worker

## 6. 缩略图缓存：基于 storage_root 配置

**决策时间**：Phase 2 开发期间

**选择**：缩略图缓存路径基于 storage_root 配置，不可写死

**原因**：
- 不同用户的 storage_root 可能不同
- 硬编码路径会导致其他用户无法使用

## 7. UI 风格：轻量工具风

**决策时间**：Phase 4 开发期间

**选择**：类似 Eagle/CleanShot 的轻量工具风格

**原因**：
- 浅灰背景 + 白色面板，视觉舒适
- 不过度设计，保持专业感
- 使用全局 QSS 样式表统一管理

## 8. 文档体系：最小可行

**决策时间**：2026-05-26

**选择**：只保留 3 个核心文档

**原因**：
- 用户是 0 编程经验，过多文档会增加负担
- CLAUDE.md 给 Claude 看（项目规范）
- README.md 给人看（如何运行）
- docs/DECISIONS.md 记录决策（避免重复讨论）
- docs/PLAN_TEMPLATE.md 计划模板（规范开发流程）

## 9. 管理员密码：PBKDF2-SHA256

**决策时间**：2026-05-29

**选择**：管理员有密码保护，普通用户无密码选择即登录

**原因**：
- 公司环境需要管理员操作（删除用户、删除索引等）有身份验证
- 普通用户仅浏览和编辑分类，无需密码
- 使用 Python 标准库 `hashlib.pbkdf2_hmac`，无额外依赖
- 密码哈希格式 `salt_hex:hash_hex`，100000 轮迭代

## 10. 用户会话：进程级内存单例

**决策时间**：2026-05-29

**选择**：`core/session.py` 用全局变量存储当前用户，不持久化

**原因**：
- 无密码体系下无需持久化登录状态
- 每次启动都要选择用户，最简方案
- 存储 id、username、role、display_name 四个字段

## 11. 局域网配置：settings.json + 配置向导

**决策时间**：2026-06-11

**选择**：首次启动弹出配置向导，输入中枢电脑 IP，保存到 `~/.image_gallery/settings.json`

**原因**：
- 同事不懂代码，不能手动设环境变量
- settings.json 是最简单的持久化方案（无需额外库）
- 局域网模式下数据库账号密码写死（`gallery` / `ImageGallery2026`），同事只需输入 IP
- 本地开发模式下仍支持环境变量，保持向后兼容
- `config.py` 优先读 settings.json，fallback 到环境变量

## 12. 引擎重建：rebuild_engine()

**决策时间**：2026-06-11

**选择**：`database.py` 新增 `rebuild_engine()` 函数，支持运行时重建数据库引擎

**原因**：
- `config.py` 在 import 时读取配置，值会冻结
- 配置向导完成后需要切换到新的数据库连接
- `rebuild_engine()` 先 `dispose()` 旧引擎，再用新配置创建新引擎
- `main.py` 中先 `importlib.reload(config)` 再 `rebuild_engine()`，确保配置生效
